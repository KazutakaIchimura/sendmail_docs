## 1. DB設計


| カラム名 | 型 | 制約 | 説明 |

#### `mail_sends`

| カラム名 | 型 | 制約 | 説明 |
|---|---|---|---|
| id | BIGINT | PK | |
| user_id | BIGINT | FK → users.id / NOT NULL | |
| office_id | BIGINT | FK → offices.id / NOT NULL | |
| send_type | ENUM('PLAN','MONITORING') | NOT NULL | |
| send_month | DATE | NOT NULL | 月初日で保存 |

| batch_id | BIGINT | FK → mail_send_batches.id / NULL可 | |
| created_by | BIGINT | FK → staffs.id | |
| created_at | TIMESTAMP | NOT NULL | |
| updated_at | TIMESTAMP | NOT NULL | |

### ステータス遷移

```
PENDING（送付待ち） → SENT（送付済み） → DONE（完了）
```

### 補足

- `send_month` は月初日で統一保存（例: 2025年5月 → `2025-05-01`）
- 月遅れ判定：`send_month < 当月の月初日 AND status = 'PENDING'`
- 重複防止：`user_id + office_id + send_type + send_month` の組み合わせはアプリ側でバリデーション

| メソッド | パス | 説明 | 権限 |
|---|---|---|---|
| GET | /api/dashboard | サマリ情報を取得 | STAFF以上 |

```json
// GET /api/dashboard レスポンス例
{
  "pendingCount": 12,
  "overdueCount": 3,
  "sentThisMonthCount": 8,
  "currentMonth": "2025-05",
  "overdueMonths": [
    { "month": "2025-03", "count": 1 },
    { "month": "2025-04", "count": 2 }
  ],
  "recentHistory": [
    {
      "id": 10, "userName": "田中太郎", "officeName": "事業所Aグループホーム",
      "sendType": "PLAN", "sentAt": "2025-05-13T10:30:00"
    }
  ]
}
```

- `pendingCount`: `status='PENDING'` の件数
- `overdueCount`: `isOverdue=true`（`send_month < 当月月初 AND status='PENDING'`）の件数
- `sentThisMonthCount`: 当月中に `status` が `SENT`/`DONE` になった件数
- `currentMonth`: 現在月（`YYYY-MM` 形式）
- `overdueMonths`: 期限超過レコードが存在する月ごとの件数一覧（`month` 昇順）。各要素は `{ "month": "YYYY-MM", "count": N }`
- `recentHistory`: 直近の送付履歴（`sent_at` 降順、最大5件）。各要素の ID フィールドは `id`（`mailSendId` ではない）
- 対象データが0件の場合も配列は空配列・件数は0で200を返す

#### `mailSendIds[]` のバリデーション・対象レコードの扱い

- `mailSendIds[]` が空配列の場合 → 400（バリデーションエラー）

```java
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/api/auth/login").permitAll()
            .requestMatchers("/api/staffs/**").hasRole("ADMIN")
            .requestMatchers("/api/**").authenticated()
            .anyRequest().permitAll()           // Reactの静的ファイル
        )

            .loginProcessingUrl("/api/auth/login")
            .successHandler(/* 200 JSON返却 */)
            .failureHandler(/* 401 JSON返却 */)

        .logout(logout -> logout

            .logoutSuccessHandler(/* 200 JSON返却 */)
        )
        .sessionManagement(session -> session
            .sessionCreationPolicy(SessionCreationPolicy.IF_REQUIRED)
        );
    return http.build();
}
```

## 4. DB初期化設計（DatabaseMigrator）

### 方針

Flywayは使用せず、アプリ起動時に `config/DatabaseMigrator.java`（Spring `@Component`）が `@PostConstruct` でマイグレーションを実行する。`schema.sql` / `data.sql` は使用しない。

### DatabaseMigrator が行う処理（起動ごと）

1. **ロール初期データ投入**: `INSERT IGNORE INTO roles (name) VALUES ('ADMIN'), ('STAFF')`
2. **`staffs.role_id` 列マイグレーション**: `role_id BIGINT` 列が未存在の場合のみ追加し、既存の `role ENUM` 列の値をもとに `role_id` を埋め、FK制約を付与する
3. **`offices.building` 列マイグレーション**: `building VARCHAR(200)` 列が未存在の場合のみ `AFTER postal_code` で追加する
4. **初期管理者作成**: `admin@example.com` が未存在の場合のみ、初期パスワード（別途安全な手段で共有）をBCryptハッシュ化して挿入する（`force_password_change=true`）

> Spring の `JdbcTemplate` を使って直接SQL実行するため、MySQL固有の `AFTER` 句などを使用できる。H2では一部DDLが `bad SQL grammar` になるが、`COLUMN_NAME` の存在確認で処理をスキップするため起動は成功する（Testcontainers＋MySQL環境での動作は `DatabaseMigrationIT` で検証済み）。

## 今後の課題

> 本セクションは2026年6月に実施した相談支援専門員によるシステムレビューに基づく課題一覧です。

### 重大度: 高（稼働前の対応を推奨）

#### アクセスログ（監査証跡）の未実装
- **背景・根拠**: 個人情報保護法第23条（安全管理措置）および厚生労働省「福祉・介護関係事業者における個人情報の適切な取扱いのためのガイダンス」により、「誰が・いつ・誰の情報を参照・更新したか」の記録が求められる
- **リスク**: 情報漏洩発生時の原因追跡・責任特定が不可能
- **対応方向**: `access_logs` テーブルの追加（操作者・対象リソース・操作種別・日時）またはアプリケーションレベルのログ記録

#### 本番環境における個人情報の国外保管リスク

> **注記**: 仕様書上の旧記述（Heroku）は実態と異なる。実際の本番環境は **Vercel（フロントエンド）＋ Railway（バックエンド・DB）**。

- **Vercel（フロントエンド）**: 静的ファイル（HTML/JS/CSS）のみをホスティングするため、個人情報はVercelのサーバーに保管されない。国外保管の問題は生じない。
- **Railway（バックエンド・DB）**: 利用者の個人情報（氏名・生年月日・備考等）はRailwayのDBに保管される。RailwayはUS企業であり、利用可能なリージョンは米国・EU・シンガポール等に限られ、日本国内リージョンは提供されていない（2025年8月時点）。
- **背景・根拠**: 個人情報保護法第24条（外国にある第三者への提供の制限）。Railwayに対して個人情報の取扱いを「委託」する場合、委託先がデータにアクセスできる立場にあるため第三者提供の例外に該当するかは慎重な判断が必要（個人情報保護委員会ガイダンスQ&A参照）
- **リスク**: 利用者（障害者）の個人情報が本人の同意なく国外のサーバーに保管されることへの法的リスク（Herokuと同水準のリスクが残る）
- **対応方向**:
  - RailwayのリージョンをEU West（Amsterdam）に設定することを優先的に検討する。EUは日本の個人情報保護委員会が認定した「同等水準国・地域」であり、第24条の同意取得義務を回避できる可能性が最も高い（ただしGDPR上の別途要件が発生するため総合的な判断が必要）
  - Railwayとのデータ処理に関する契約（DPA等）を締結し、委託先監督義務（同法第25条）を履行すること
  - 事業所の個人情報保護方針・プライバシーポリシーに、外国のクラウド事業者への委託に関する記載を追加すること
  - 詳細は個人情報保護の専門家（弁護士・認定個人情報保護団体等）への確認を推奨

#### 要配慮個人情報としての管理方針が未策定
- **背景・根拠**: 利用者（障害者）の情報は個人情報保護法第2条第3項の「要配慮個人情報」（「心身の機能の障害があること」）に該当し、通常の個人情報より厳格な管理が求められる。取得・第三者提供には本人の明示的な同意が必要（同法第20条第2項・第27条第2項）
- **リスク**: 法令上の義務違反となる可能性がある
- **対応方向**: 設計書への要配慮個人情報としての位置づけ明記、プライバシーポリシー・利用規約の整備、本人同意取得フローの設計

#### 担当相談支援専門員と利用者の紐付けがない
- **背景・根拠**: 実務では利用者ごとに担当の相談支援専門員が割り当てられるが、現在のDB設計では全スタッフが全利用者情報に一律アクセスできる
- **リスク**: 担当外スタッフによる不必要な個人情報閲覧が発生しうる。情報管理の責任所在が不明確
- **対応方向**: `user_staff` テーブル（担当紐付け）またはusersテーブルへの `assigned_staff_id` フィールドの追加。ロールベースアクセス制御との組み合わせで担当者のみ参照可能にすることを検討

#### モニタリングサイクル管理機能がない
- **背景・根拠**: 障害者総合支援法・指定相談支援の指定基準（平成24年厚生労働省令第28号）により、モニタリングは利用者ごとに実施頻度（毎月・3ヶ月・6ヶ月等）が設定される。新規利用者・重度障害者はより高頻度が求められる場合がある
- **リスク**: 現状では「今月モニタリング対象の利用者」を自動で割り出す機能がなく、やり忘れによる指定基準違反が起きうる
- **対応方向**: `monitoring_cycles` テーブルの追加（利用者ごとの周期・次回予定日・担当スタッフを管理）。ダッシュボードに「今月のモニタリング対象者一覧」を追加

#### 記録保存ポリシーが未定義
- **背景・根拠**: 指定相談支援事業者の指定基準（障害者総合支援法第51条の20等に基づく省令）では業務帳票類の5年間保存義務がある。一方、個人情報保護法上は利用目的達成後の遅滞ない消去も求められる
- **リスク**: 保存期間・廃棄方針が不明確なままでは法令上の義務を果たせない可能性がある
- **対応方向**: データ保存期間（最低5年）・廃棄方針の運用規程整備および設計書への明記
