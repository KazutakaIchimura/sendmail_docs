## 1. DB設計

### ER図（概要）

```
staffs ────────────────────────────────┐
                                       │ created_by / sent_by
offices（事業所）                      │
   │                                   │
   │ 多対多                            │
   │                                   ↓
users（利用者）          mail_sends（送付レコード）
   │                         │
   └── user_offices ──────────┘
       （中間テーブル）       │
                              └── mail_send_batches（送付バッチ）
```

### テーブル定義

#### `staffs`

| カラム名 | 型 | 制約 | 説明 |
|---|---|---|---|
| id | BIGINT | PK | |
| name | VARCHAR(100) | NOT NULL | 氏名 |
| email | VARCHAR(255) | UNIQUE / NOT NULL | ログインID |
| password_hash | VARCHAR(255) | NOT NULL | |
| role | ENUM('ADMIN','STAFF') | NOT NULL | |
| is_active | BOOLEAN | DEFAULT true | |
| force_password_change | BOOLEAN | DEFAULT true | 初回変更フラグ |
| created_at | TIMESTAMP | NOT NULL | |
| updated_at | TIMESTAMP | NOT NULL | |

#### `users`

| カラム名 | 型 | 制約 | 説明 |
|---|---|---|---|
| id | BIGINT | PK | |
| name | VARCHAR(100) | NOT NULL | 氏名 |
| name_kana | VARCHAR(100) | | ふりがな |
| birth_date | DATE | | 生年月日 |
| notes | TEXT | | 備考 |
| is_active | BOOLEAN | DEFAULT true | |
| created_at | TIMESTAMP | NOT NULL | |
| updated_at | TIMESTAMP | NOT NULL | |

#### `offices`

| カラム名 | 型 | 制約 | 説明 |
|---|---|---|---|
| id | BIGINT | PK | |
| name | VARCHAR(200) | NOT NULL | 事業所名 |
| postal_code | VARCHAR(8) | | 郵便番号（`NNN-NNNN` 形式） |
| address | VARCHAR(500) | | 住所 |
| phone | VARCHAR(20) | | 電話番号 |
| is_active | BOOLEAN | DEFAULT true | |
| created_at | TIMESTAMP | NOT NULL | |
| updated_at | TIMESTAMP | NOT NULL | |

> `postal_code` はリクエストDTOで `^\d{3}-\d{4}$`（例: `123-4567`）の形式バリデーションを行う。形式不正時は400バリデーションエラー。

#### `user_offices`

| カラム名 | 型 | 制約 | 説明 |
|---|---|---|---|
| id | BIGINT | PK | |
| user_id | BIGINT | FK → users.id | |
| office_id | BIGINT | FK → offices.id | |
| created_at | TIMESTAMP | NOT NULL | |

> `user_id` + `office_id` にUNIQUE制約

#### `mail_send_batches`

| カラム名 | 型 | 制約 | 説明 |
|---|---|---|---|
| id | BIGINT | PK | |
| sent_by | BIGINT | FK → staffs.id | 送付処理者 |
| sent_at | TIMESTAMP | NOT NULL | 送付日時 |
| notes | TEXT | | メモ（任意） |
| created_at | TIMESTAMP | NOT NULL | |

> `updated_count`（更新件数）はカラムとして保持しない。`GET /api/mail-send-batches/{id}` のレスポンスでは `mail_sends` を `batch_id` で `COUNT` した値を都度算出して返す。

#### `mail_sends`

| カラム名 | 型 | 制約 | 説明 |
|---|---|---|---|
| id | BIGINT | PK | |
| user_id | BIGINT | FK → users.id / NOT NULL | |
| office_id | BIGINT | FK → offices.id / NOT NULL | |
| send_type | ENUM('PLAN','MONITORING') | NOT NULL | |
| send_month | DATE | NOT NULL | 月初日で保存 |
| status | ENUM('PENDING','SENT','DONE') | DEFAULT 'PENDING' | |
| batch_id | BIGINT | FK → mail_send_batches.id / NULL可 | |
| created_by | BIGINT | FK → staffs.id | |
| created_at | TIMESTAMP | NOT NULL | |
| updated_at | TIMESTAMP | NOT NULL | |

### ステータス遷移

```
PENDING（送付待ち） → SENT（送付済み） → DONE（完了）
```

### 補足

- `send_month` は月初日で統一保存（例: 2025年5月 → `2025-05-01`）
- 月遅れ判定：`send_month < 当月の月初日 AND status = 'PENDING'`
- 重複防止：`user_id + office_id + send_type + send_month` の組み合わせはアプリ側でバリデーション

## 2. APIエンドポイント設計

### 基本方針

| 項目 | 内容 |
|---|---|
| ベースURL | `/api` |
| 認証方式 | セッション認証（Spring Security標準） |
| レスポンス形式 | JSON |
| 権限 | 全エンドポイントにログイン必須（/api/auth/loginを除く） |

### 9.1 認証

| メソッド | パス | 説明 | 権限 |
|---|---|---|---|
| POST | /api/auth/login | ログイン・セッション発行 | 不要 |
| POST | /api/auth/logout | ログアウト・セッション破棄 | STAFF以上 |

### 9.2 ダッシュボード

| メソッド | パス | 説明 | 権限 |
|---|---|---|---|
| GET | /api/dashboard | サマリ情報を取得 | STAFF以上 |

```json
// GET /api/dashboard レスポンス例
{
  "pendingCount": 12,
  "overdueCount": 3,
  "sentThisMonthCount": 8,
  "overdueMonths": ["2025-03", "2025-04"],
  "recentHistory": [
    {
      "mailSendId": 10, "userName": "田中太郎", "officeName": "事業所Aグループホーム",
      "sendType": "PLAN", "sentAt": "2025-05-13T10:30:00"
    }
  ]
}
```

- `pendingCount`: `status='PENDING'` の件数
- `overdueCount`: `isOverdue=true`（`send_month < 当月月初 AND status='PENDING'`）の件数
- `sentThisMonthCount`: 当月中に `status` が `SENT`/`DONE` になった件数
- `overdueMonths`: 期限超過レコードが存在する `send_month` の一覧（`send_month` 昇順、重複除去）
- `recentHistory`: 直近の送付履歴（`sent_at` 降順、最大5件）
- 対象データが0件の場合も配列は空配列・件数は0で200を返す

### 9.3 利用者管理

| メソッド | パス | 説明 | 権限 |
|---|---|---|---|
| GET | /api/users | 一覧取得 | STAFF以上 |
| POST | /api/users | 新規登録 | STAFF以上 |
| GET | /api/users/{id} | 詳細取得 | STAFF以上 |
| PUT | /api/users/{id} | 更新 | STAFF以上 |
| DELETE | /api/users/{id} | 無効化（論理削除） | ADMIN |

### 9.4 利用者↔事業所紐付け

| メソッド | パス | 説明 | 権限 |
|---|---|---|---|
| GET | /api/users/{userId}/offices | 紐付き事業所一覧 | STAFF以上 |
| POST | /api/users/{userId}/offices | 事業所を紐付け | STAFF以上 |
| DELETE | /api/users/{userId}/offices/{officeId} | 紐付け解除 | STAFF以上 |

### 9.5 事業所管理

| メソッド | パス | 説明 | 権限 |
|---|---|---|---|
| GET | /api/offices | 一覧取得 | STAFF以上 |
| POST | /api/offices | 新規登録 | STAFF以上 |

| PUT | /api/offices/{id} | 更新 | STAFF以上 |

### 9.6 送付レコード

| メソッド | パス | 説明 | 権限 |
|---|---|---|---|
| GET | /api/mail-sends | 一覧取得 | STAFF以上 |
| GET | /api/mail-sends/by-office | 事業所別グループ取得 | STAFF以上 |
| POST | /api/mail-sends | 新規登録 | STAFF以上 |
| PUT | /api/mail-sends/{id} | 更新 | STAFF以上 |
| DELETE | /api/mail-sends/{id} | 削除 | ADMIN |

```json
// GET /api/mail-sends/by-office レスポンス例
[
  {
    "office": { "id": 1, "name": "事業所Aグループホーム", "address": "東京都○○区..." },
    "mailSends": [
      {
        "id": 10, "userId": 1, "userName": "田中太郎",
        "sendMonth": "2025-04", "sendType": "PLAN",
        "status": "PENDING", "isOverdue": true
      }
    ]
  }
]
```

> 重複防止チェック（`user_id` + `office_id` + `send_type` + `send_month` の組み合わせ）に違反した場合は `DuplicateResourceException` を送出し、**409** を返す（アプリ側バリデーションだが「重複」を表すため400ではなく409を用いる）。

### 9.7 送付バッチ

| メソッド | パス | 説明 | 権限 |
|---|---|---|---|
| POST | /api/mail-send-batches | まとめて送付済みに更新 | STAFF以上 |
| GET | /api/mail-send-batches/{id} | バッチ詳細取得 | STAFF以上 |

```json
// リクエスト
{ "mailSendIds": [10, 11, 12], "notes": "5月定例送付" }
// レスポンス
{ "batchId": 5, "sentAt": "2025-05-13T10:30:00", "updatedCount": 3 }

// GET /api/mail-send-batches/{id} レスポンス例
{ "batchId": 5, "sentBy": 1, "sentAt": "2025-05-13T10:30:00", "notes": "5月定例送付", "updatedCount": 3 }
```

#### `mailSendIds[]` のバリデーション・対象レコードの扱い

- `mailSendIds[]` が空配列の場合 → 400（バリデーションエラー）
- 存在しない `mailSendId` を含む場合 → 404（`ResourceNotFoundException`）
- 対象レコードに `status=PENDING` 以外（`SENT`/`DONE`）が含まれる場合 → **そのレコードは更新対象から除外し、`updatedCount` には含めない**（エラーとはせず処理は継続する）

### 9.8 スタッフ管理（ADMIN限定）

| メソッド | パス | 説明 | 権限 |
|---|---|---|---|
| GET | /api/staffs | 一覧取得 | ADMIN |
| POST | /api/staffs | 新規登録 | ADMIN |
| PUT | /api/staffs/{id} | 更新 | ADMIN |
| DELETE | /api/staffs/{id} | 無効化 | ADMIN |

#### スタッフ安全設計（DELETE /api/staffs/{id} のサーバー側バリデーション）

フロントエンド（`StaffListPage` の `canDisable()`）だけでなく、**バックエンド（`StaffService`）でも同様のチェックを行う**。

- **自己無効化禁止**: ログイン中のスタッフ自身を無効化しようとした場合はエラーとする（400）
- **ADMIN最低1名維持**: 無効化対象が ADMIN かつ、無効化により有効な ADMIN が0名になる場合はエラーとする（400）

いずれも業務ルール違反のため `MethodArgumentNotValidException` ではなく、専用の例外（例: `IllegalStateException` を `GlobalExceptionHandler` で400に変換）として扱う。

### 共通エラーレスポンス

| HTTPステータス | 用途 |
|---|---|
| 200 | 成功 |
| 201 | 作成成功 |
| 400 | バリデーションエラー |
| 401 | 未認証 |
| 403 | 権限なし |
| 404 | リソースなし |
| 409 | 重複（`DuplicateResourceException`。一意制約違反・メールアドレス重複・送付レコードの重複登録など） |
| 500 | サーバーエラー |

### SecurityConfig（概要）

```java
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/api/auth/login").permitAll()
            .requestMatchers("/api/staffs/**").hasRole("ADMIN")
            .requestMatchers("/api/**").authenticated()
            .anyRequest().permitAll()           // Reactの静的ファイル
        )
        .formLogin(form -> form
            .loginProcessingUrl("/api/auth/login")
            .successHandler(/* 200 JSON返却 */)
            .failureHandler(/* 401 JSON返却 */)
        )
        .logout(logout -> logout
            .logoutUrl("/api/auth/logout")
            .logoutSuccessHandler(/* 200 JSON返却 */)
        )
        .sessionManagement(session -> session
            .sessionCreationPolicy(SessionCreationPolicy.IF_REQUIRED)
        );
    return http.build();
}
```

## 4. DB初期化設計（schema.sql / data.sql）

### 方針

Flywayは使用せず、Spring Bootの標準機能である `schema.sql` / `data.sql` でDB初期化を行う。
環境ごとにDDLが異なる（MySQL vs PostgreSQL）ため、platformファイルで管理する。

### data.sql（初期データ・共通）

```sql
-- 初期管理者（パスワード: "changeme" のBCryptハッシュ・初回ログイン時に変更必須）
INSERT INTO staffs (name, email, password_hash, role, is_active, force_password_change)
SELECT '管理者', 'admin@example.com',
    '$2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy',
    'ADMIN', TRUE, TRUE
WHERE NOT EXISTS (
    SELECT 1 FROM staffs WHERE email = 'admin@example.com'
);
```

> `WHERE NOT EXISTS` で2回目の起動時の重複INSERTを防ぐ。

## 5. テスト環境設計（バックエンド）

### 5.1 テスト戦略（テストピラミッド）

```
          ▲
         /E2E\          少数・重要フローのみ
        /------\        （Playwright）
       /  統合  \       APIレイヤー・DB結合
      /----------\      （Spring Boot Test）
     /  ユニット  \     ビジネスロジックの検証
    /--------------\    （JUnit5 + Mockito）
```

---

### 5.2 バックエンド テスト

#### テストライブラリ

| ライブラリ | 用途 |
|---|---|
| JUnit5 | テストフレームワーク |
| Mockito | モック・スタブ |
| MockMvc | Controllerテスト |
| AssertJ | アサーション（流暢なAPI） |

#### レイヤー別テスト方針

| レイヤー | テスト種別 | 使用ツール | DB |
|---|---|---|---|
| Service | ユニットテスト | JUnit5 + Mockito | 不要（モック） |
| Controller | MVCテスト | MockMvc | モック（Serviceをモック） |

> Controllerテストは Service をモックするため、パスワードのハッシュ化・`send_month` の正規化・`batch_id`/`sent_by` の設定・スタッフ安全設計（自己無効化禁止／ADMIN最低1名維持）など**永続化やビジネスロジックを伴うアサーションは Service／Repository レイヤーのテストで検証する**。Controllerテストではリクエスト/レスポンスの形式・ステータスコード・権限制御の検証に範囲を限定する。
