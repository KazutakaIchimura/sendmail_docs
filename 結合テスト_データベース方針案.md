# 結合テストにおけるデータベース方針案

作成日: 2026-06-21
更新日: 2026-06-22（案B（Testcontainers + 実MySQL）を実装。**ステータス: 導入ステップ1〜4 実施済み**）

## 実装状況（2026-06-22）

案Bを実装した。

- `pom.xml`: `org.testcontainers:testcontainers-junit-jupiter` / `testcontainers-mysql`、`org.springframework.boot:spring-boot-testcontainers` を test スコープで追加。`maven-failsafe-plugin` を追加し、`mvn test`（Surefire）は単体テストのみ・`mvn verify`（Failsafe）で `*IT` クラスも実行されるよう分離した。
- `src/test/resources/application-it.properties`: 結合テスト用プロファイル（`it`）。DataSource接続情報は `@ServiceConnection` がコンテナ情報で上書きするため未指定、`ddl-auto=none` + MySQLDialect のみ上書き。
- `src/test/java/.../integration/`:
  - `AbstractIntegrationTest` — `@Testcontainers` + 静的 `@Container @ServiceConnection MySQLContainer`（mysql:8.0）。サブクラス間でコンテナ1個を共有。
  - `DatabaseMigrationIT` — 実MySQL上でアプリを起動し、`DatabaseMigrator` が完走すること（roles投入・初期管理者作成・staffs.role_id列・offices.building列の最終状態）を確認。
  - `MySqlLegacySchemaMigrationIT` — Springコンテキストを使わず、旧スキーマ（`staffs.role` ENUM列・`offices`にbuilding列なし）を素のJDBCで再現してから `DatabaseMigrator` と同一のALTER文（`ADD COLUMN ... AFTER` 等）を実行し、実MySQL上で成功することを直接検証。H2で出ていた `bad SQL grammar` 警告に対する回帰テスト。
  - `MailSendCrudIT` — mail-sends の作成→一覧→更新→削除、重複検出を実MySQL（ENUM型カラムを含む）上で確認。
- CI（`backend/.github/workflows/ci.yml`）: `./mvnw test` → `./mvnw verify` に変更。GitHub Actions の `ubuntu-latest` には標準でDockerが入っているため追加設定なしでTestcontainersが動作する想定。

**未実施（要対応）:**
- ローカル開発者の Docker Desktop 有無の確認（導入ステップ5の前半）

**解決済み（2026-06-23）:**
- 結合テスト用MySQLのバージョンをRailwayの実バージョンと揃えた。Railway CLI（`railway status`）で本番MySQLサービス（`MySQL-VnPq` / `MySQL`）のイメージを確認した結果、いずれも `mysql:9.4` だったため、`AbstractIntegrationTest` / `MySqlLegacySchemaMigrationIT` の `MySQLContainer` 指定を `mysql:8.0` → `mysql:9.4` に変更した。
- GitHub Actions上で `./mvnw verify` を実際に実行して動作確認した（導入以来、初めての実機確認）。その結果、`MailSendCrudIT` 実行時に `Connection refused` で失敗するバグを発見・修正した。
  - 原因: `AbstractIntegrationTest` に `@Testcontainers` + `@Container` を付けていたため、JUnit5拡張が **サブクラスごとに** `beforeAll` で `MYSQL.start()` を呼び出していた。同じ `static` フィールドでも呼び出しごとにコンテナが再作成され（コンテナID・公開ポートが変わる）、`DatabaseMigrationIT` が使っていたコンテナが捨てられた状態になり、`MailSendCrudIT` 実行時に古い接続情報で繋ぎにいって `Connection refused` になっていた。
  - 修正: `AbstractIntegrationTest` から `@Testcontainers` / `@Container` を外し、`static` イニシャライザで明示的に1回だけ `MYSQL.start()` を呼ぶ「Singleton container」パターンに変更（`@ServiceConnection` は維持）。これによりサブクラス間で同一コンテナが安全に共有される。

## 背景

現状の単体テスト（`MailSendControllerTest` 等）は `@SpringBootTest` + H2 インメモリDB（`jdbc:h2:mem:testdb`）で実行している。
テスト実行ログ上、以下の警告が出ており、H2 と本番 MySQL の間に SQL 方言の差異があることが確認できている。

```
WARN ... DatabaseMigrator : Migration: staffs role column migration skipped:
StatementCallback; bad SQL grammar [ALTER TABLE staffs ADD COLUMN role_id BIGINT NULL]
WARN ... DatabaseMigrator : Migration: building column skipped:
StatementCallback; bad SQL grammar [ALTER TABLE offices ADD COLUMN building VARCHAR(200) NULL AFTER postal_code]
```

`ADD COLUMN ... AFTER` は MySQL 固有構文で H2 ではサポートされない。つまり **H2 上で全テストが green でも、本番 MySQL 環境で同じマイグレーション・クエリが正しく動く保証にはならない**。結合テストでこの種の差異を検出できる構成にしておく必要がある。

本番構成（[[project_deployment.md]] 参照）:
- バックエンド: Railway 上の Spring Boot（MySQL接続）
- DB: Railway MySQL（`MySQL-VnPq` / `MySQL`）

## 比較した3つの選択肢

### 案A：H2 インメモリのまま（現状維持）

| 項目 | 評価 |
|---|---|
| 速度 | ◎ 非常に速い、CIへの追加コストなし |
| 本番再現度 | ✗ MySQL固有構文・型・ロック挙動を検証できない |
| 導入コスト | ◎ ゼロ（既存のまま） |

**結論**: 単体テスト（サービス層のロジック検証）には適しているが、結合テストの目的（本番DBとの整合性確認）には不十分。既に方言差異による警告が出ている時点でリスクが顕在化している。

### 案B：Testcontainers + 実MySQLコンテナ（推奨）

`@Testcontainers` でテスト実行時に Docker 上に MySQL コンテナを立て、`@DynamicPropertySource` で接続情報を注入する方式。

| 項目 | 評価 |
|---|---|
| 速度 | △ コンテナ起動分のオーバーヘッド（数秒〜十数秒/クラス） |
| 本番再現度 | ◎ 本番と同じ MySQL バージョン・方言で検証できる |
| 導入コスト | △ Docker が必要（ローカル・CI実行環境の両方） |
| 分離性 | ◎ テストごとに独立したクリーンなDBインスタンス |

**実装イメージ:**
```java
@SpringBootTest
@Testcontainers
class MailSendIntegrationTest {

    @Container
    static MySQLContainer<?> mysql = new MySQLContainer<>("mysql:9.4");  // 本番Railwayと同バージョン

    @DynamicPropertySource
    static void registerProps(DynamicPropertyRegistry registry) {
        registry.add("spring.datasource.url", mysql::getJdbcUrl);
        registry.add("spring.datasource.username", mysql::getUsername);
        registry.add("spring.datasource.password", mysql::getPassword);
    }

    // ...
}
```

**メリット:**
- `DatabaseMigrator` の `ALTER TABLE ... AFTER` のような MySQL 固有処理を正しく検証できる
- 本番と同条件でトランザクション・ロック・文字コード（utf8mb4等）の挙動を確認できる
- コンテナは毎回使い捨てなので、テスト間のデータ汚染がない

**デメリット・対応策:**
- ローカル開発者全員に Docker Desktop（または同等）が必要 → 既に多くのチームで標準化されている前提
- GitHub Actions では `services:` または Testcontainers の Docker-in-Docker サポートで動作可能（追加設定が必要）
- 単体テスト全部をこの方式に置き換えると実行時間が伸びる → **案Cとのハイブリッドを推奨**

### 案C：CI上でのMySQLサービスコンテナ（GitHub Actions `services:`）

Testcontainers ライブラリを使わず、GitHub Actions のワークフロー側で MySQL サービスコンテナを直接起動し、固定のホスト/ポートで接続する方式。

| 項目 | 評価 |
|---|---|
| 速度 | ○ ジョブ全体で1コンテナを共有するため、Testcontainers より起動回数が少ない |
| 本番再現度 | ◎ 案Bと同等 |
| ローカル実行 | △ ローカルでは別途 `docker-compose up` 等の手動起動が必要（Testcontainersのような自動起動はない） |

```yaml
services:
  mysql:
    image: mysql:8.0
    env:
      MYSQL_ROOT_PASSWORD: test
      MYSQL_DATABASE: sendmail_test
    ports:
      - 3306:3306
```

## 推奨方針

**単体テスト（既存）と結合テストを分離した二段構成にする。**

| テスト種別 | DB | 目的 | 配置 |
|---|---|---|---|
| 単体テスト | H2 インメモリ（現状維持） | サービス層のロジック・コントローラーのI/O検証を高速に回す | `src/test/java/.../controller/*Test.java`, `service/*Test.java` |
| 結合テスト | Testcontainers + MySQL（案B） | マイグレーション・MySQL固有SQL・実DBとの整合性を検証 | `src/test/java/.../integration/*IT.java`（新規パッケージ） |

**理由:**
1. 既存の H2 ベース単体テスト（2026-06-22時点で110件）は資産として活かせる。書き直しは不要。
2. 結合テストは「本番DBに近い環境で何が壊れるか」を検出する役割に特化させ、テスト数を絞る（CRUD一連の正常系＋マイグレーション確認など、重要シナリオ中心）。
3. Maven の `Failsafe` プラグイン（または単純にクラス名サフィックス `*IT` で区別）を使い、`mvn test` では単体テストのみ、`mvn verify` で結合テストも含めて実行する分離が一般的。

**導入ステップ（案）:**
1. `pom.xml` に Testcontainers 依存追加（`testcontainers`, `mysql`, `junit-jupiter` モジュール）
2. `src/test/java/com/example/sendmail/integration/` パッケージを新設
3. `DatabaseMigrator` が実行する `ALTER TABLE` 系の処理が実際の MySQL 上で成功するかを確認するテストを最優先で追加（現状 H2 で `bad SQL grammar` 警告が出ている箇所）
4. 主要エンドポイント（mail-sends の CRUD・CSV出力、users/staffs/offices の登録〜無効化フロー）を結合テストとしてカバー
5. CI（GitHub Actions）に Docker 実行環境があることを確認し、Testcontainers が動作するようワークフローを調整
